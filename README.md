### 🛠️ Technical Rigor & Architecture: The Modeling Pipeline

The engine follows a rigorous four-stage transformation process to ensure every "What-If" scenario is mathematically grounded:

1.  [cite_start]**High-Dimensional Clustering (Segmentation):** Using `scikit-learn` **K-Means**, the engine identifies latent customer personas [cite: 3, 20, 39, 83-90]. [cite_start]This moves the analysis beyond simple demographic filters to behavioral "runways," allowing for high-resolution segmented lifecycles [cite: 3, 10-11, 143-153].

2.  [cite_start]**Survival Modeling (The Statistical Core):** The system implements the **Cox Proportional Hazards (CPH)** model[cite: 3, 19, 51]. [cite_start]Unlike standard regression, this handles "right-censored" data—customers who haven't had the event yet—ensuring future LTV is not underestimated [cite: 3, 35-37].

3.  [cite_start]**Hazard Shift Simulation (The ROI Logic):** The `run_scenario_dashboard` function applies proportional hazard multipliers to specific features [cite: 3, 106-134]. [cite_start]By shifting the baseline hazard, the engine calculates the **Delta in Median Survival Time**, which is then converted into the **Defended LTV** metric [cite: 3, 13, 125-133].

4.  [cite_start]**Automated Artifact Generation:** Using `python-pptx` and `reportlab`, the engine bypasses manual reporting [cite: 3, 20, 171-192]. [cite_start]It maps internal dataframes directly to "Executive-Ready" slides, ensuring the **Forensic Rationale** from the audit ledger is never separated from the strategic recommendation [cite: 3, 41-42, 172-182].
